# Lohko Development

## Use this skill when

- The task touches Lohko plugin source in `/home/maxemilian/projects/jcore-lohko`.
- The user asks to create or modify JCORE WordPress blocks.
- The user asks about block rendering, interactivity, or Twig templates.
- The current repository contains `jcore.toml` and uses Lohko for block implementations.

## Repository map

- `lohko.php`: plugin bootstrap, block registration, mime additions, translation setup.
- `includes/timber.php`: shared Timber context helpers for block rendering.
- `src/<block>/block.json`: block metadata and asset wiring.
- `src/<block>/index.js` + `edit.js`: editor registration and edit UI.
- `src/<block>/render.php` + `view.twig`: dynamic block rendering via Timber/Twig.
- `src/<block>/view.js`: frontend behavior for interactive blocks.
- `build/`: compiled assets and `blocks-manifest.php` generated by `@wordpress/scripts`.

## Development workflow

1. Install dependencies: `pnpm install`.
2. Build once: `pnpm build` or watch with `pnpm watch`.
3. For local WP test environment, use `.wp-env.json` with `pnpm env:start`.
4. Validate code quality: `pnpm lint:js`, `pnpm lint:css`, and `pnpm format`.
5. If changing block metadata/assets, verify generated `build/*` outputs are in sync.

## Block implementation pattern

1. Define block contract in `src/<block>/block.json`.
2. Register editor behavior in `src/<block>/index.js` and `edit.js`.
3. For dynamic blocks, set `save: () => null` and implement `render.php`.
4. Render markup through Twig templates and shared Timber context helpers.
5. For interactivity, use `viewScriptModule` and `@wordpress/interactivity` in `view.js`.

## Registration and compatibility notes

- Lohko uses `blocks-manifest.php` registration for performance.
- In modern WordPress versions it prefers `wp_register_block_types_from_metadata_collection`.
- It includes fallback logic for environments that only support older metadata APIs.
- Keep registration changes centralized in `lohko.php` rather than per-block hacks.

## Communication guidance

- Distinguish source-of-truth files (`src/`) from generated artifacts (`build/`).
- Mention whether a fix is editor-only, frontend-only, or both.
- When a change depends on WordPress version behavior, call that out explicitly.
